# Know Where PRD

## 1. 제품 개요

### 1.1 제품명
- Know Where

### 1.2 한 줄 정의
- URL, 요약, 태그, 메모를 빠르게 저장하고 다시 찾기 위한 개인용 지식 카드 앱

### 1.3 제품 목표
- 사용자가 웹에서 본 자료를 10초 안에 저장할 수 있어야 한다.
- 저장된 자료를 검색과 폴더 탐색으로 즉시 다시 찾을 수 있어야 한다.
- 카드 상세 모달에서 핵심 요약, 원문 링크, 개인 메모를 한 번에 확인할 수 있어야 한다.
- 저장 직후 메타데이터 수집과 썸네일 생성이 비동기로 진행되어도 UI는 즉시 반응해야 한다.

### 1.4 제품 형태
- 단일 사용자용 로컬 웹앱
- 프론트엔드: React + TypeScript + Vite
- 백엔드: Django + Django REST Framework
- DB: SQLite
- 파일 저장: 로컬 파일 시스템(media)

## 2. 문제 정의

사용자는 브라우저 북마크만으로는 다음 문제를 해결하기 어렵다.

- 북마크에 메모와 요약을 구조적으로 남기기 어렵다.
- 저장된 링크를 나중에 검색으로 찾기 어렵다.
- 링크의 맥락이 사라져 왜 저장했는지 기억하기 어렵다.
- URL만 저장하면 제목, 설명, 썸네일 등 부가 정보가 부족하다.

Know Where는 이 문제를 카드 기반 지식 저장 방식으로 해결한다.

### 3.1 핵심 사용자

### 3.2 사용 환경

- 상시 온라인을 전제로 하지 않음

## 4. 핵심 가치 제안

- URL 저장이 빠르다.
- 저장 후 자동으로 제목과 일부 메타데이터를 채운다.
- 카드 단위로 요약, 태그, 메모를 함께 관리한다.
- 검색 결과가 즉시 반영된다.
- 썸네일과 상세 모달로 기억 회복 비용을 줄인다.

## 6. 성공 지표

### 6.1 정량 지표
- URL 저장 요청 후 카드가 목록에 보이기까지 1초 이내
- 검색 입력 후 결과 반영 200ms 이내
- 카드 목록 첫 렌더 1.5초 이내
- 카드 생성 성공률 99% 이상
- 썸네일 생성 성공률 80% 이상

### 6.2 정성 지표
- 사용자가 저장 후 “왜 저장했는지” 메모로 회상 가능해야 한다.
- 저장/탐색/수정의 핵심 흐름에서 별도 설명 없이 사용할 수 있어야 한다.

## 7. 핵심 사용자 시나리오

### 7.1 URL 저장
1. 사용자가 상단 입력창에 URL을 입력한다.
2. 필요하면 폴더를 선택한다.
3. 저장 버튼을 누른다.
4. 프론트는 즉시 optimistic card를 목록 상단에 표시한다.
5. 백엔드는 URL 유효성 검증 후 카드를 생성한다.
6. 제목이 비어 있으면 페이지 title을 수집한다.
7. 썸네일 생성 작업을 백그라운드로 시작한다.
8. 프론트는 카드 상태를 `processing`에서 `ready` 또는 `failed`로 갱신한다.

### 7.2 검색
1. 사용자가 검색창에 문자열을 입력한다.
2. 프론트는 debounce 후 API를 호출한다.
3. 백엔드는 제목, 태그, 메모, URL 기준으로 부분 일치 검색한다.
4. 최신순으로 결과를 반환한다.
5. 프론트는 현재 폴더 조건과 함께 결과를 반영한다.

### 7.3 상세 확인 및 메모 수정
1. 사용자가 카드 클릭
2. 상세 모달 오픈
3. 제목, URL, 태그, 요약, 상세정보, 메모, 생성일, 썸네일 상태 확인
4. 메모 수정 후 저장
5. 백엔드는 patch 업데이트
6. 프론트 목록과 상세 모달 데이터를 동시에 동기화

### 7.4 폴더 관리
1. 사용자가 새 폴더 생성
2. 카드 생성 시 폴더 지정
3. 폴더 클릭 시 목록 필터링
4. 폴더 삭제 시 카드 처리 정책 선택
5. 기본 정책: 카드들은 `미분류` 폴더로 이동

## 8. 기능 요구사항

### 8.1 카드
- URL은 필수
- title, summary, details, memo는 선택 입력 가능
- tags는 문자열 배열
- folder는 1개만 지정
- 카드 상태 필드 필요
- 썸네일 상태 필드 필요

### 8.2 제목 처리
- 사용자가 직접 입력 가능
- 비어 있으면 서버가 `<title>` 자동 수집
- 수집 실패 시 도메인 기반 fallback 제목 생성
- 예: `react.dev 문서`

### 8.3 요약/세부내용
- 초기 버전에서는 사용자가 직접 입력하거나 빈 값 허용
- 후속 버전에서 AI 요약 파이프라인 확장 가능하도록 필드 유지

### 8.4 메모
- 카드 상세에서 빠르게 수정 가능
- 빈 문자열 허용
- 목록에서 메모 존재 여부를 표시

### 8.5 태그
- 쉼표 입력 또는 태그 입력 UI 지원
- 중복 태그 제거
- 대소문자 및 공백 normalize

### 8.6 폴더
- 기본 폴더 필요: `전체`, `미분류`
- 사용자 정의 폴더 생성 가능
- 정렬 순서 저장 가능

### 8.7 썸네일
- 카드 생성 직후 백그라운드 작업 시작
- 상태값: `pending | processing | ready | failed`
- ready 시 이미지 경로 저장
- 실패해도 카드 저장은 성공으로 처리

### 8.8 검색 및 정렬
- 단일 검색창
- 검색 대상: title, url, memo, tags
- 기본 정렬: `created_at desc`
- 추후 정렬 확장을 위해 `sort` 파라미터 구조는 미리 설계

## 9. 비기능 요구사항

### 9.1 성능
- 목록 API는 페이지네이션 지원
- 상세 API는 필요 시 별도 호출
- 썸네일 이미지 lazy loading

### 9.2 안정성
- 썸네일 실패가 전체 카드 생성 실패로 이어지면 안 된다.
- 메타데이터 수집 실패 시 fallback 데이터로 저장 가능해야 한다.

### 9.3 유지보수성
- 프론트와 백엔드 계약은 OpenAPI 수준으로 명시 가능해야 한다.
- 백엔드 비동기 작업은 인터페이스로 추상화해 작업 방식 교체가 쉬워야 한다.

### 9.4 보안
- 외부 인증은 없지만 URL 입력값 검증 필요
- 서버측 SSRF 위험을 줄이기 위한 허용 정책 필요
- 로컬 환경이라도 private IP 대역 접근 차단 권장

## 10. 시스템 아키텍처

### 10.1 프론트엔드 구조
- React + TypeScript + Vite
- UI 상태와 서버 상태를 분리
- 서버 상태 관리는 TanStack Query 권장
- 라우팅은 React Router
- 스타일은 Tailwind CSS 또는 CSS Modules 중 택1
- 현재 목업 구조를 고려하면 Tailwind CSS가 구현 속도와 일치성이 좋음

### 10.2 백엔드 구조
- Django
- Django REST Framework
- 앱 분리
- `cards`: 카드/태그/검색
- `folders`: 폴더 관리
- `ingestion`: 메타데이터 수집
- `thumbnails`: 썸네일 생성
- `core`: 설정, 공통 유틸, 예외 처리

### 10.3 비동기 처리 전략
- 1차 버전 최적안: Django 내부 작업 큐 + DB 기반 job table
- 이유:
- 로컬 단일 사용자 앱에 Celery/Redis는 과하다.
- SQLite 기반에서도 구현 가능하다.
- 실패 재시도와 상태 추적이 가능하다.

### 10.4 향후 확장
- 작업량 증가 시 Celery + Redis 또는 Dramatiq로 교체 가능
- 추상 인터페이스는 유지

## 12. UI 설계 고정 기준

이 섹션은 현재 제공된 목업 UI를 구현 기준안으로 고정하기 위한 문서다.  
개발 과정에서 기능 구현 편의를 이유로 레이아웃, 정보 구조, 주요 상호작용을 임의 변경해서는 안 된다.

### 12.1 UI 설계 목적
- 현재 목업의 정보 구조를 실제 서비스 UI의 기준안으로 확정한다.
- 프론트 구현 과정에서 개발자 판단으로 화면 구조를 바꾸지 못하도록 제약한다.
- 추후 UI 변경이 필요하면 코드 수정 전에 먼저 본 문서를 개정해야 한다.

### 12.2 화면 구성 원칙
- 메인 화면은 단일 페이지 구조를 유지한다.
- 레이아웃은 `상단 헤더 + 좌측 폴더 사이드바 + 우측 카드 콘텐츠 영역 + 상세 모달` 구조를 유지한다.
- 카드 상세는 별도 페이지로 전환하지 않고 모달로 표시한다.
- URL 입력과 검색 입력은 모두 상단 헤더 영역에 유지한다.
- 폴더 탐색은 좌측 사이드바에 유지한다.
- 카드 목록은 우측 메인 영역의 카드 그리드 구조를 유지한다.

### 12.3 메인 레이아웃 고정안

```text
[Header]
  - 좌측: 서비스 로고/브랜드
  - 중앙: URL 입력 + 저장 버튼, 검색 입력
  - 우측: 알림, 설정, 사용자 영역

[Body]
  - 좌측 Sidebar: 폴더 목록
  - 우측 Main: 폴더 제목, 정렬/뷰 토글, 카드 그리드

[Overlay]
  - 카드 클릭 시 상세 모달 오픈
```

### 12.4 헤더 설계 고정 기준

#### 필수 유지 요소
- 브랜드 영역
- URL 입력창
- URL 실행 버튼
- 검색 입력창
- 우측 유틸 아이콘 영역

#### 배치 규칙
- 브랜드는 좌측 고정
- URL 입력과 검색 입력은 중앙 배치
- 알림/설정/사용자 아이콘은 우측 고정

#### 변경 금지
- URL 입력창을 사이드바나 모달로 이동 금지
- 검색창을 메인 콘텐츠 내부로 이동 금지
- 헤더 전체를 2단 구조로 분해 금지

#### 허용 변경
- 버튼 문구 변경
- 아이콘 교체
- 미세한 spacing 조정
- 반응형에서 입력창 폭 조정

### 12.5 좌측 폴더 사이드바 고정 기준

#### 필수 유지 요소
- `내 폴더` 제목
- 폴더 추가 버튼
- 폴더 목록
- 선택된 폴더 강조 상태
- 활성 폴더 카드 수 표시

#### 동작 규칙
- 폴더 클릭 시 메인 목록 필터링
- 전체 폴더는 항상 최상단 유지
- 시스템 폴더와 사용자 폴더를 같은 시각적 계층에서 보여주되, 삭제 가능 여부는 기능적으로 구분

#### 변경 금지
- 폴더 기능을 드롭다운 하나로 축소 금지
- 사이드바를 기본 상태에서 숨김 처리 금지
- 폴더 목록을 상단 탭 구조로 대체 금지

### 12.6 메인 콘텐츠 영역 고정 기준

#### 상단 정보 바
- 현재 선택 폴더명 표시
- 현재 결과 수 표시
- 정렬 컨트롤 유지
- 그리드/리스트 뷰 토글 유지

#### 카드 목록 영역
- 기본 뷰는 카드 그리드
- 카드 간 여백과 독립된 블록 감각 유지
- 비어 있는 상태에서는 empty state 화면 유지
- 분석/처리 중 카드는 skeleton 카드로 표시

#### 변경 금지
- 목록을 기본 테이블 UI로 대체 금지
- 카드 클릭 동작을 hover-only 인터랙션으로 축소 금지
- 결과 수 문구 제거 금지

## 15. 카드 생성 처리 흐름

1. 프론트 `POST /api/cards`
2. 백엔드 URL 검증 및 normalize
3. DB에 카드 저장
4. title 비어 있으면 metadata fetch job 등록
5. thumbnail job 등록
6. API는 즉시 카드 응답 반환
7. worker가 작업 수행
8. 완료 시 카드 상태 갱신
9. 프론트는 polling 또는 revalidation으로 상태 반영

## 22. 최적 기술 선택 결론

이 제품의 현재 요구사항 기준 최적안은 다음이다.

- 프론트: React + TypeScript + Vite + TanStack Query + React Router + Tailwind CSS
- 백엔드: Django + DRF + SQLite + Playwright
- 비동기: DB 기반 lightweight worker

이 조합이 최적인 이유는 다음과 같다.

- 로컬 단일 사용자 앱이라 인프라 복잡도를 최소화할 수 있다.
- 현재 카드 기반 UI와 REST API 구조가 잘 맞는다.
- React Query가 optimistic update, refetch, polling 요구사항에 적합하다.
- Django/DRF는 CRUD, serializer, admin, 파일 저장, SQLite 활용에 강하다.
- Playwright는 썸네일 생성 정확도가 높다.
- Celery/Redis 없이도 초반 요구사항을 충분히 충족한다.
